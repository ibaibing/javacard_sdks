    1. Introduction

      2. The Java CardTM 2.2 Development Kit tools run on a workstation using a Java Virtual Machine (VM). The Java Card 2.2 Virtual Machine is written in the "C" programming language. Separate bundles for the Solaris® and Windows NT® versions are available for this release. For more details on the reference implementation, please refer to the User's Guide that is installed with the Java Card 2.2 Development Kit.

      The issues discussed in these Release Notes pertain to both the Java Card 2.2_01 Development Kit Binary Release and the Java Card 2.2_01 Development Kit Binary Release with Cryptography Extensions.

      Issues that are marked (Development Kit with Cryptography Extensions only) pertain only to the Java Card 2.2_01 Development Kit Binary Release with Cryptography Extensions.

      Version 2.2_01 Update Release Information

      The Java Card version 2.2_01 Update Release corrects an error in the original version 2.2 API specification of the signature of the javacard.security.KeyAgreement.generateSecret method. As a result of the update, the content of the following files have changes since the original Java Card Development Kit version 2.2 release:

      • KeyAgreement.java—the KeyAgreement class with the corrected signature for generateSecret. The contents of this source file have changed for both the cryptographic extensions release and non-cryptographic extensions release.
      • api.jar—jar file that contains the Java Card API. The contents of this file have changed for both the cryptographic extensions release and non-cryptographic extensions release.
      • security.exp—corrected export file for the javacard.security package. This package contains the KeyAgreement class.

       

      The following file in the Java Card 2.2 Specification download has also been modified :

      • JavaCard22API.pdf—the corrected version of the API specification for Java Card 2.2. The API specification is now "Revision 1.1".

       

      Updating an Existing Java Card 2.2 Release

      If you have downloaded Java Card version 2.2_01 only to update an existing Java Card version 2.2 release, you must copy the files listed below to their appropriate locations in the file system. Note that the 2.2_01 release retains the same directory structure as the 2.2 release.

      In the paths listed below, the UNIX path separator "/" is used. If you are replacing files on the Windows platform, replace "/" with "\". All instances of the files, especially security.exp, must be replaced.

      • KeyAgreement.java—If you are using the source release, copy this file to the java_card_kit-2_2/src/share/Java/api/javacard/security directory. If you are using the source release with cryptographic extensions, you must also copy this file to java_card_kit-2_2/src/withcrypto/share/java/api/javacard/security.
      • api.jar—Copy the 2.2_01 version of this file to the java_card_kit-2_2/lib directory.
      • security.exp—Copy the 2.2_01 version of this file to the java_card_kit-2_2/api_export_files/javacard/security/javacard/ directory.
      • JavaCard22API.pdf—Copy the 2.2_01 version of this file to the jcspecs2.2/specs directory.

    2. New Features and Improvements

      • Object Deletion Mechanism

        • Object Deletion Mechanism feature lets you reclaim memory which is being used by "unreachable" objects.

      • Package Deletion

        • Package deletion lets you remove all of a package's code from the card's memory. This feature is not supported in JCWDE.

      • Applet Deletion

        • Applet deletion lets you delete up to eight applets simultaneously, including all of their child objects, from Java Card's memory. This feature is not supported in JCWDE.

      • Remote Method Invocation

        • You can write RMI applications for the Java Card platform. You can develop RMI applets that run on the card and RMI client applications that run on a Card Acceptance Device (CAD) terminal which supports a J2SE or J2ME platform. This feature is not supported in JCWDE.

      • Logical Channel Support

        • This release of Java Card 2.2 reference implementation has the ability to support up to three logical channels. This gives an ISO 7816-4-compliant terminal the ability to open up to three sessions into a smart card using the reference implementation, one session per logical channel.

      • Conversion Verification
         

        • The converter now invokes the verifier to verify the export files it reads in, and also checks for the validity of the CAP file and export file it outputs. If the CAP file being generated is inconsistent, it reports appropriate errors and does not actually generate the CAP file.

        • There is a -noverify command line option that allows it to bypass verification.

      • Cryptography Support (Development Kit with Cryptography Extensions only)

        • Java Card 2.2 provides an implementation of basic security and cryptography classes. Both C-JCRE and JCWDE support the following encryption algorithm implementations:

        • MessageDigest — SHA-1 (SHA)
        • Random Data — pseudo random number generator

        Signature and Cipher support for RSA and DES with the following implementation details:

        • Key Length — 64-bit DES, 128-bit DES3, 512-bit RSA
        • Signature — DES MAC8 Signature with ISO 9797 M2 Padder,
          RSA Signature with SHA Message Digest and PKCS1 encoding format
        • Cipher — DES Cipher Block Chaining with ISO 9797 M2 Padder,
          RSA Cipher with PKCS1 encoding format

    3. Documentation Roadmap

      4. Documentation for the Java Card 2.2 Development Kit can be found in the Development Kit product download and the Specification download.

      These documents (in PDF format) are included in the product download:

      • Java CardTM 2.2 Development Kit User's Guide describes the reference implementation. It is targeted at developers who are creating applets using the Java Card 2.2 Application Programming Interface (Sun Microsystems, Inc., 2002), and also at developers who are considering creating a vendor-specific framework based on the Java Card 2.2 technology specifications.

      • Java CardTM 2.2 Application Programming Notes contains tips and guidelines for the applet developer who is using the new features of Java Card 2.2 such as applet and package deletion, object deletion, logical channels, and RMI. It is also for developers who are considering creating a vendor-specific framework based on the Java Card 2.2 technology specifications.

      • Java CardTM 2.2 RMI Client Application Programming Interface describes the Java Card RMI client-side architecture and API.

      • Java CardTM 2.2 Off-Card Verifier describes the Java Card 2.2 Off-Card Verifier. The Off-Card Verifier provides functionality for verifying CAP files and export files. When applied to the set of CAP files that will reside on a Java Card-compliant smartcard and to the set of export files used to construct those CAP files, the Off-Card Verifier provides the means to assert that the content of the smartcard has been verified.

      These documents (in PDF format) are included in the specification download:

      • Java CardTM 2.2 Application Programming Interface, Revision 1.1 provides the documentation for the Java Card API.

      • Java CardTM 2.2 Virtual Machine Specification assists developers in creating an implementation, developing a specification to extend the Java Card technology specifications, or in creating an extension to the Java Card Runtime Environment (JCRE).

      • Java CardTM 2.2 Runtime Environment (JCRE) Specification is intended to assist JCRE implementors in creating an implementation, developing a specification to extend the Java Card technology specifications, or in creating an extension to the Java Card Runtime Environment (JCRE).

    4. Known Bugs and Issues

      5. This section describes software and documentation limitations.

      Static Blank Finals Limitations

      Static blank finals cannot be initialized in initialization blocks. These initializations must be inlined.

      Object Deletion Mechanism Limitations

      The number of objects that can be deleted using the Object Deletion Mechanism in the C-JCRE is restricted by the amount of transaction space available. In case Object Deletion Mechanism is invoked and there are more objects to be deleted than can be handled by the deletion mechanism, the implementation will throw an error and terminate the card session.

      Applet/Package Deletion Limitations

      Since Applet and Package Deletion in the C-JCRE depend on the Object Deletion Mechanism, the limitation on Object Deletion Mechanism may also affect these features.
